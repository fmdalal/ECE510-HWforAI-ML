# Analysis

### Why is the naive kernel memory-bound?
The number of FLOPs used in Naive and Tiled method are the same as the total number of computation remains constant, however, what changes is how the data is accessed and that comes down to the number of FLOPs per byte of DRAM traffic (arithmetic intensity). Considering the roofline model, the values on the left of the ridge point indicate the kernel is memory-bound and on the right is considered compute-bound. This logically makes sense as if our kernel is in the memory-bound region and we instead of increasing its memory bandwidth capability we increase the computational FLOPs which would have no effect on the memory-bound kernel.

### How tiling reduces DRAM traffic?
When doing tiling, the matrices is broken into smaller matrices. These smaller matrices are used by many of the threads to do partial computation simultaneously in a coordinated manner with the matrices being stored in shared memory . In the naive kernel, each thread independently issues its own global loads for its own row of A and column of B. Thus, the coordination of fetchhing data between the threads is missing. This reduces DRAM traffic by a factor between T (the realistic reduction achieved by TILE-sized shared memory blocks) and N (the theoretical upper bound if every element were loaded exactly once). For our TILE=8 implementation, the reduction is 8× — the lower end of this range.

### Whether your tiled kernel achieved the expected improvement - If not identify the remaining bottleneck.
My tiled kernel showed a Speedup of 1.54 times (considering the data from nvcc compilation which shows the wall-clock time). This was not the expected improvement as the expectation was for the tiled kernel to be on the diagonal of the memory-bound region of the roofline plot. This was theoretically the expectataion as the compute capability of the kernel has not improved but the data accessing was an improvement. The key reason is specific to the setup being used and our matrix size. At N=1024 on a T4 processor, the 4 MB matrices fit in the 4 MB L2 cache, so the naive kernel's "redundant" DRAM loads are mostly absorbed by cache (as seen in nsight profiler report - 98% L2 hit rate, DRAM at only 13 GB/s). Tiling's DRAM-reduction advantage is largely claimed by L2. At larger N, or with a bigger tile size, the speedup would be much more dramatic.  
However, since the tiled kernel improvement is still within the memory-bound region to the left of the ridge point in the roofline model and not bandwidth or compute saturated the bottleneck identified is latency bound. Both kernels have saturated memory pipelines (99% L1/TEX throughput for naive, 98% for tiled) with only 2 warps per block at 8×8. There aren't enough warps in flight per SM to hide load-use latency.
